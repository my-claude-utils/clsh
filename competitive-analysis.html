Lessons for clsh

What Paseo and Claude Cockpit can teach us about running Claude Code reliably from an Android phone

The Landscape

Three Philosophies

Same problem space, very different bets on what matters.

Analogy: Vehicles

Cockpit is a workstation with multiple monitors — powerful but bolted to a desk. Paseo is a fully loaded SUV with every option — comfortable but heavy to maintain. clsh is a motorcycle — minimal, fast to start, goes anywhere. The question is: which SUV features belong on the motorcycle?

Current Advantages

What clsh Already Does Better

Don't fix what isn't broken. These are genuine differentiators.

✓

4-Tier Tunnel Fallback

ngrok → Tailscale → SSH → Wi-Fi. Paseo has a single relay. Cockpit is localhost-only. clsh works from anywhere, with real fallbacks.

✓

tmux Session Persistence

Real session survival across agent restarts, not PID recovery (Cockpit) or JSON snapshots (Paseo). Your Claude keeps running even if clsh crashes.

✓

WebAuthn Biometric Auth

Fingerprint to unlock, no passwords. Cockpit has no auth at all. Paseo uses ECDH key pairing — secure but not as ergonomic on mobile.

✓

Zero-Install Web Client

Scan a QR code and go. No App Store, no APK sideload, no Expo build. This is the motorcycle advantage — instant start.

Ranked Recommendations

Lessons Worth Stealing

Ordered by impact-per-effort. Click to expand rationale and details.

1
Aggressive Terminal Re-fitting
Fix the #1 Android annoyance: keyboard show/hide breaks the terminal
Small effort Paseo
Analogy

Android's virtual keyboard is like a cat walking on your desk — it rearranges everything unpredictably. Paseo's solution is to just keep putting everything back every few milliseconds until the cat leaves.

What Paseo does:

  • Staggered fit timeouts: 0, 16, 48, 120, 250, 500, 1000, 2000ms after mount
  • Periodic 250ms fit interval to catch missed resize events
  • Force re-fit on visibilitychange (critical for app resume on Android)
  • visualViewport resize listener (keyboard show/hide)
  • -webkit-text-size-adjust: 100% to prevent browser font scaling
  • overscroll-behavior: none to kill pull-to-refresh
Why #1

Highest impact-to-effort ratio. This is mostly CSS and a few event listeners — maybe 2 hours of work — but it fixes the most common "it feels broken on my phone" moments. The terminal not resizing when the keyboard opens/closes is the kind of paper cut that makes people give up.

2
Session State Tracking
Know which Claude sessions need your attention without reading output
Medium effort Cockpit
Analogy

Right now, checking on your Claude sessions is like peeking through keyholes — you have to look at each one. State tracking gives you indicator lights above each door: green (done), yellow (thinking), red (needs you).

What Cockpit does: Strips ANSI from PTY output, runs regex against Claude's known prompt patterns, classifies sessions as idle, busy, waiting for input, or starting. Uses a 10-second debounce to avoid false positives during Claude's thinking pauses.

What clsh has today: Basic idle detection (IDLE_THRESHOLD = 2500ms) based on output activity, but no semantic understanding of what the session is doing, and no UI exposure.

What to build: Parse output for Claude's prompt markers (the $ / > after it finishes, the "Thinking..." indicator). Surface state as colored dots on the session grid. Push notifications when a session transitions to "waiting."

Why #2

On a phone, context switching between sessions has real cost — you lose your scroll position, the keyboard might re-trigger, etc. Knowing which session actually needs you before tapping into it saves enormous friction. This is the biggest "quality of life" improvement for multi-session workflows on mobile.

3
Server-Side Terminal Snapshots
Instant reconnection instead of replaying thousands of buffer entries
Medium effort Paseo
Analogy

Current reconnection is like rewatching a movie from the beginning to find where you left off. Server-side snapshots are like a bookmark — you open right to the current page.

What Paseo does: Runs @xterm/headless server-side, mirrors all PTY output into it, then extracts the full terminal grid (cells with fg/bg colors, attributes, cursor position, scrollback) as a structured snapshot. On reconnect, the client receives one snapshot instead of replaying history.

What clsh does today: Buffers up to 50,000 entries per session and replays them on reconnect. On a flaky mobile connection that drops and reconnects frequently, this means potentially replaying megabytes of ANSI data each time.

Implementation: Add @xterm/headless as a server dependency. Feed PTY output to the headless terminal in parallel with the WebSocket. On client connect/reconnect, call extractGrid() and send a single snapshot message. Delta streaming after that.

Why #3

Mobile connections drop. A lot. Every reconnection that takes 3 seconds instead of 200ms erodes trust in the tool. This also enables a future feature: multiple clients viewing the same session simultaneously (e.g., pair programming from two phones). The effort is real but contained — the server-side change is isolated to PTY output handling.

4
Voice Input (Web Speech API)
Dictate prompts instead of thumb-typing on glass
Small MVP Paseo
Analogy

Typing a paragraph-long prompt to Claude on a phone keyboard is like eating soup with a fork — technically possible, but there's a better utensil right there. Android Chrome supports the Web Speech API natively.

What Paseo does: Full STT pipeline with three backends (Sherpa-ONNX local, Deepgram, OpenAI Realtime), custom Expo audio modules, and bidirectional TTS. Massive investment.

What clsh could do as MVP: A single microphone button that uses the browser's built-in SpeechRecognition API. Transcribed text goes into the terminal input. Works in Chrome on Android today, zero server changes. Maybe 50 lines of code.

A more advanced version could detect when Claude is waiting for input and auto-focus the dictation, or add a "speak to Claude" mode that wraps the transcribed text in a prompt format.

Why #4

The MVP is nearly free — browser API, no server changes, no dependencies. But it's ranked below state tracking and snapshots because those fix reliability issues, while voice is an ergonomics upgrade. Reliability first, then comfort. That said, for the "on the go" use case (walking, commuting), voice input could be transformative.

5
MCP Orchestrator
Let one Claude command the others — your phone becomes a control tower
Large effort Cockpit
Analogy

Right now clsh gives you walkie-talkies to talk to each Claude individually. The MCP orchestrator gives you an air traffic controller who talks to all of them on your behalf — you just tell the controller what you want.

What Cockpit does: An MCP server script exposes 5 tools (list_sessions, send_input, get_output, get_state, create_session). One Claude session is designated as "orchestrator" and gets these tools via --mcp-config-path. It can spawn workers, send them instructions, and poll their output.

Why it fits clsh: clsh already has multi-session infrastructure — session CRUD, WebSocket routing, PTY management. The MCP server would be a thin wrapper over existing APIs. The big unlock: from your phone, you tell one Claude "refactor the auth module, run tests in another session, and fix any failures" — and it coordinates everything while you walk to the coffee shop.

Scope: MCP server binary (~200 lines), Claude config generation, UI for designating orchestrator sessions. Requires #2 (state tracking) to work well.

Why #5

This is the feature that changes what clsh is — from a remote terminal to a remote command center. But it's ranked 5th because it depends on state tracking (#2) and benefits from snapshots (#3), so the foundations should come first. It's also the largest single investment. Save this for when the mobile basics are rock-solid.

6
E2E Encryption
Application-layer crypto so no tunnel provider can read your traffic
Medium effort Paseo
Analogy

TLS through ngrok is like mailing a sealed letter through a courier — the courier promises not to open it. E2EE is like putting the letter in a lockbox where only you and the recipient have the key. The courier couldn't read it even if they wanted to.

What Paseo does: ECDH key exchange via NaCl/TweetNaCl during pairing. The QR code includes the server's public key. After handshake, all WebSocket frames are encrypted with XSalsa20-Poly1305 using the derived shared key. The Cloudflare relay only sees ciphertext.

What clsh could do: Embed server's ECDH public key in the QR URL. Client generates keypair, sends its public key in the first WebSocket message. Derive shared key, encrypt all subsequent frames. Uses the Web Crypto API on the client side (no extra dependencies).

Why #6

Nice-to-have, not need-to-have. With Tailscale (WireGuard) or SSH tunnels, clsh already has strong transport encryption. E2EE matters most on the ngrok path, where you're trusting a third party. Ranked last because it doesn't improve the day-to-day mobile UX — it's a defense-in-depth layer. Worth building once everything else is solid.

What NOT to Adopt

Equally important: knowing which features would pull clsh off its axis.

Analogy

A motorcycle that tries to become an SUV just becomes a bad SUV. These features would add weight without serving the "grab your phone, fix the thing" mission.

×
Native app (Expo/RN)
Web-only is a feature. No App Store approval, no install step, no update lag. The QR-to-terminal flow is clsh's killer UX.
×
Electron desktop
Not the target use case. If you're at your desk, just use the terminal directly.
×
Multi-provider support
Paseo supports Claude + Codex + OpenCode. clsh is Claude-focused — that simplicity is a feature, not a gap.
×
JSON file persistence
SQLite + tmux is strictly more robust than Paseo's JSON approach. Don't downgrade.
